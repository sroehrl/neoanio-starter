Useful CLI Commands

The CLI is based on Symfony Console.

Command

Description

php cli list
Lists all available commands
php cli list:routes
Lists all registered routes
php cli create:controller [name]
Creates a Routable class based on qualified name (example: App\Controller\MyController)
php cli create:controller [name]
Creates a Routable class based on qualified name (example: App\Controller\MyController)
php cli create:model [name]
Creates a model based on qualified name (example: App\Model\MyModel)
php cli migrate:model [dialect] [name]
Synchronizes a model with your database based on dialect (example: sqlite) and qualified name (example: App\Model\MyModel)
About Styling

This starter project ships with glass, a Gaudiamus-CSS-based design.
Ultimately, it serves as a demonstration only and can be completely replaced. If you want to play around with it, though, feel free to explore the root-folder scss. For your convenience the file compile.js is setup to compile the scss files into your public css-directory.
NOTE: the framework requires dart-sass, making compilation a little slow.

To use it:
yarn install or npm install

Then:
node compile.js

About Authentication

In all examples in setup API-examples were used. The login is set up to register a stateful (Session, based on josantonius/php-session ) and a stateless (JWT, based on neoan3-apps/stateless ) authentication. Depending on your needs, the following middleware is available

Middleware

Description

App\Middleware\NeedsAuth
Requires a valid stateless authentication as a bearer-token in the Authorization header. Attaches the decoded JWT to the $provided-array as "auth" on success and throws a 401 (unauthorized) error when authorization isn't successful.
App\Middleware\NeedsSession
Requires a valid userId in the session. Attaches the valid user's id to the $provided-array as "auth" on success and throws the custom Error Reauthenticate which redirects the user to the login page (make changes as you need).
About SSR apps

As mentioned before, other then resolving visitable endpoints, calls are made as API-calls instead of e.g. making post-calls to a view. In order to do this as convenient as possible, this project ships with some native javascript code in the skeleton. It's purpose is to enable an almost HTML-only way of sending requests to the API. Let's have a look:

<form method="get" action="Sapi/auth/me" then="myMethod">
  <button type="submit">Am I logged in?</button>
</form>

<script>
  function myMethod(result){
    console.log(result);
  }
</script>

What happens here? Javascript intercepts the native GET-submission and transforms it into an axios-call using your JWT token (if logged in). In order to remain in control of interactivity, we added the attribute then to enable execution of custom javascript code (or one of the included functions like "reloadPage") as well as catch to react on errors.
